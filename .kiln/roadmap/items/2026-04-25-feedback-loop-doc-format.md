  ## Shape (locked 2026-04-25)

  Reuse wheel workflow JSON as the on-disk format for feedback-loop
  definitions, with a reserved `_meta:` extension carrying loop-specific
  semantics that wheel ignores at runtime. Same artifact serves three
  audiences:

    - Agents (parse JSON for loop structure)
    - Humans (rendered markdown with embedded Mermaid diagram)
    - Validators (wheel JSON schema gives free correctness checks)

  **Shape A — dual-file (locked):**

      docs/feedback-loop/<name>.json   ← canonical, wheel-shaped, validatable
      docs/feedback-loop/<name>.md     ← generated by render-workflow.sh

  v1 is documentation-only — files live under `docs/feedback-loop/*.json`
  and are NEVER passed to `wheel-run`. The runnable-vs-doc distinction is
  enforced by convention via `_meta.kind: feedback-loop` filter. The
  rendered `.md` carries a "do not edit by hand — regenerate via
  render-workflow.sh" banner at the top.

  **Why `docs/feedback-loop/` (not `.kiln/feedback-loops/`):**
  `.kiln/` is for runtime state (issues, mistakes, in-flight roadmap).
  Feedback loops are documentation, meant to be read — `docs/` matches
  semantics. Singular `feedback-loop/` per user preference (sibling
  directories like `docs/features/` use plural; this one diverges
  intentionally).

  ## `_meta:` schema (v1, locked 2026-04-25)

  ### Top-level (loop-wide)

      {
        "name": "retro-to-pi-apply",
        "description": "<one-line — wheel's existing field>",
        "_meta": {
          "kind": "feedback-loop",          // REQUIRED — discriminator
          "status": "active",                // REQUIRED — active | proposed | retired
          "owner": "kiln-core",              // REQUIRED — accountability

          "triggers": [                      // optional — what kicks the loop off
            "github label:retrospective",
            "manual /kiln:kiln-pi-apply"
          ],

          "metrics": {                       // optional — open key/value
            "latency_target": "1 week",
            "current_p50": "3 weeks",
            "throughput": "2-3 retros/month"
          },

          "anti_patterns": [                 // optional — mistakes to avoid
            "auto-applying retros without human review"
          ],

          "related_loops": [                 // optional — cross-references
            "capture-to-distill"
          ],

          "last_audited": "2026-04-25"       // optional — ISO date
        },
        "steps": [...]
      }

  ### Per-step (annotations)

      {
        "id": "fetch-retro-issues",
        "type": "command",
        "command": "...",
        "_meta": {
          "actor": "agent",                  // optional — agent | human | hook | command
          "doc": "Pulls every open retro-labeled issue from gh. Skipped if zero retros — downstream steps no-op.",
          "notes": [                         // optional — additional bullets
            "currently scoped to one repo; will need rework for multi-repo"
          ]
        }
      }

  ### Field rationale

  - `kind` — filterability via `jq '._meta.kind == "feedback-loop"'`. Required for the convention to work at all.
  - `status` — lets us mark loops retired without deleting them. Documentation of past loops is valuable.
  - `owner` — accountability. If it breaks, we know who's looking.
  - `triggers` — most useful answer to "when does this happen?"; surfaces in rendered metadata.
  - `metrics` — open key/value because every loop measures different things. Structured schema would force false consistency.
  - `anti_patterns` — distilled wisdom that's easy to lose. Same shape as the existing `_meta` field on roadmap items.
  - `related_loops` — lets the rendered page link sideways. Cheap to maintain.
  - `last_audited` — freshness indicator. Loops rot; this surfaces stale ones.
  - `actor` (per-step) — closed enum (`agent | human | hook | command`) so the renderer can tag/color steps consistently.
  - `doc` (per-step) — multi-line markdown. Biggest contributor to readable rendered output.

  ### Deliberately NOT included

  - `failure_modes` — overlaps with `anti_patterns`. Pick one (anti-patterns wins; more actionable).
  - `cadence` — overlaps with `triggers` + `metrics.throughput`. Redundant.
  - `success_signal` — vague; `metrics` already covers this concretely.
  - `outputs` — wheel workflows already encode outputs via step `output:` fields. Don't duplicate.
  - `purpose` — wheel's existing top-level `description:` field covers this.
  - `rendered_at` (was in earlier sketch) — convention is `<input>.md` next to `<input>.json`; no need to encode the path.

  ### Validation posture

  No schema enforcement in v1. Wheel's permissive-on-unknown-keys posture means `_meta:` validates by-construction. Document the shape here; if drift becomes a real problem, add a sidecar validator in v2 — same propose-don't-apply pattern as `/kiln:kiln-claude-audit`.

  ### Open question (revisit after seed loops)

  `metrics` keys are totally open in v1 (no curated initial set), because we don't have enough loops yet to know what the consistent cross-loop set should be. After three seed loops author, see if a curated subset emerges; if so, document it here as the "common metrics" convention without making it a hard requirement.

  ## Renderer (BUILT 2026-04-25)

  Lives at `plugin-wheel/scripts/render/render-workflow.sh` — owned by
  wheel, not kiln, because it operates on wheel workflow JSON
  regardless of whether the input is runnable or conceptual. ~120 lines
  of bash + jq. No skill wrapper yet (deferred — direct script
  invocation is fine for v1).

  Behavior:

    1. Reads the JSON file.
    2. Walks `steps[]` + `context_from:` edges (wheel's actual edge
       field — `depends_on:` is not used in current schema). Falls
       back to sequential edges when no `context_from:` anywhere.
    3. Emits Mermaid `flowchart TD` syntax.
    4. Wraps in markdown with: title + description, "do not edit"
       banner, metadata table (kind/status/owner/metrics from
       `_meta:`), anti-patterns + related-loops sections (when
       present), Mermaid diagram, per-step prose sections (one per
       step, drawn from `_meta.doc` first then falling back to the
       command/instruction body).
    5. Default output: `docs/feedback-loop/<basename>.md`. Override
       via second positional arg.

  Smoke-tested against `plugin-kiln/workflows/kiln-report-issue.json`
  — DAG (parallel branch from `create-issue`) rendered correctly.

  GitHub renders Mermaid natively in markdown so the same output file IS
  the HTML render path — no separate static-site build needed for v1.
  Optional v2: a small static-site step (Astro / Eleventy / `marked`)
  that bundles all rendered loops into a docs site.

  ## Initial seed loops (proves the format — NOT YET AUTHORED)

  These are the v1 deliverables. Each one exercises a different
  property of the format; together they pressure-test the `_meta:`
  schema before it ossifies.

    a. `retro-to-pi-apply.json` — already mostly built (the
       /kiln:kiln-pi-apply skill drives this); this just documents it
       as a loop with metrics.

    b. `capture-to-distill.json` — issue/feedback → roadmap promotion
       → distill into PRD. Spans multiple skills + state transitions.
       Good test of cross-skill loop semantics.

    c. `build-prd-pipeline.json` — specify → plan → tasks → implement
       → audit → PR → retro. Documents the full pipeline as a loop
       (retro feeds back into next PRD intake).

  Three loops should make the format earn its keep — fewer means we
  haven't pressure-tested the schema.

  ## What this format buys

    - **Single source of truth** — one JSON file, three consumers.
    - **Validatable** — wheel JSON schema catches structural mistakes
      free.
    - **Mermaid for free** — `steps[]` + `depends_on:` is already a
      DAG. Renderer is ~50 lines of jq.
    - **Composable** — loops can reference other loops via
      `_meta.related_loops` + a future `_meta.calls_loop:` field.
    - **Future-runnable** — if a loop turns out to map cleanly to
      execution, drop the `_meta.kind` filter and it runs in wheel
      with no schema change.

  ## Tradeoff (acknowledge)

  Conceptual-only means wheel never executes them. The "this is
  runnable JSON but never run" smell is real — could mislead a future
  contributor into trying `wheel-run retro-to-pi-apply` and getting
  surprised. Mitigations:

    - Different directory (`.kiln/feedback-loops/` vs `workflows/`)
    - Required `_meta.kind: feedback-loop` field
    - `wheel-list` skips files without `_meta.kind: workflow` (or
      whatever the runnable kind ends up being)
    - The renderer surfaces "DOCUMENTATION ONLY — not runnable" in
      the rendered output

  ## Out of scope for v1

  - Actually running these (would require wheel to gain
    `--allow-conceptual` or similar)
  - Per-step duration/latency tracking
  - Cross-loop graph rendering (a loop-of-loops view)
  - Auto-extraction from existing skills/workflows (would derive
    a draft loop JSON from a wheel workflow file)
  - Web-rendered interactive diagrams (D3 / Cytoscape)
  - `/wheel:wheel-render` skill wrapper around the script
  - PostToolUse hook to auto-render on JSON edit (sibling to the
    `compile-agents.sh` hook pattern)
  - CI drift check (`md == render(json)`)

  All v2 candidates once the v1 format proves its keep.

  ## Success signal

  When asked "what feedback loops does kiln have?", the answer is
  "look at `.kiln/feedback-loops/`" — and the rendered markdown is
  good enough to onboard a new contributor without further prose.

# Feedback-loop documentation format — wheel-workflow-JSON-shaped, conceptual-only

## What

Define a documentation-only format for feedback loops that reuses
wheel workflow JSON as the schema, augmented with a reserved `_meta:`
block for loop-specific semantics. v1 files live under
`docs/feedback-loop/*.json`, never get `wheel-run`. The render script
(`plugin-wheel/scripts/render/render-workflow.sh` — built 2026-04-25)
emits markdown with an embedded Mermaid diagram derived from
`steps[]` + `context_from:`, plus per-step prose annotations from
`_meta.doc`. Three seed loops (retro→PI-apply, capture→distill,
build-prd pipeline) prove the format's keep.

## Why now

The kiln stack has many feedback loops in practice (retro→action,
capture→roadmap→PRD, QA→implementer fix loop) but no canonical place
to point at when explaining them. New contributors reverse-engineer
the loops from skills and commit history. A definable format that's
machine-readable, human-renderable, and validatable closes the
observability gap without committing to runnable infrastructure
yet.

## Why this shape (vs. alternatives)

YAML frontmatter + markdown body would also work, but reusing wheel
JSON gives validation for free, makes the format obviously composable
(loops are graphs of steps, same as workflows), and leaves the door
open to making loops runnable later without a format change.

The `_meta:` extension is the load-bearing trick — it lets us add
loop semantics (owner, metrics, anti-patterns) without polluting
wheel's schema, and wheel's permissive-on-unknown-keys posture means
files validate cleanly today.

## Hardest part

Picking the `_meta:` shape so it's stable. Once seeded loops cite
fields, retroactive renames cost real work. Mitigation: ship with
the conservative initial set (kind, status, owner, metrics,
anti_patterns, related_loops), document the extension policy
(additive only, no renames), and accept v2 may add fields.

## Cheaper version

Skip the renderer in v1 — author the JSON files, but rely on humans
reading raw JSON until N≥3 loops demand a tool. Saves the render
script work but gives up the "human-readable" leg of the
three-audience promise. Recommended only if scope pressure hits;
the renderer is ~50 lines of jq + a markdown wrapper, so cheap to
ship together.

## Acceptance signal

A new contributor asks "what feedback loops does kiln have?", we
point at `.kiln/feedback-loops/_rendered/`, and they get a one-page
overview per loop with a diagram, owner, current metrics, and
per-step explanation — enough to understand without further prose.
