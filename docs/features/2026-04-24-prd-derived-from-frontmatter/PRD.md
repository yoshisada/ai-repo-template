# Feature PRD: PRD `derived_from:` Frontmatter

**Date**: 2026-04-24
**Status**: Draft
**Parent PRD**: [docs/PRD.md](../../PRD.md) (placeholder; product context inherited from `CLAUDE.md`)

## Background

Every PRD in this plugin is derived from one or more `.kiln/issues/` or `.kiln/feedback/` source items, but that derivation is written only as a human-readable `### Source Issues` markdown table in the PRD body. The reverse direction is already machine-readable — when `/kiln:kiln-distill` bundles items, it writes `prd: <path>` into each source file's frontmatter. The forward direction (PRD → sources) is not. Any consumer that wants to answer "which items fed this PRD?" has to either parse the body table with a markdown-aware regex or invert the one-to-many `prd:` map by scanning every backlog file.

The architectural concern raised by this feedback is that the missing frontmatter field is the root cause of a recurring class of bug. `/kiln:kiln-build-prd` Step 4b's job is "given PRD path, flip every item derived from it to `completed`." With no `derived_from:` on the PRD, Step 4b inverts the map by string-matching `prd:` values across every backlog file — which is exactly the path-normalization failure mode that PR #146 just shipped a fix for (feedback-side directory skipped, trailing-slash / relative-vs-absolute comparisons, diagnostic-on-zero-matches safety belt). A `derived_from:` list on the PRD eliminates the matching problem at its source: Step 4b reads the PRD and iterates the list. No matching, no normalization, no diagnostic needed. The same simplification applies to `/kiln:kiln-hygiene`'s `merged-prd-not-archived` rule, which today performs the same reverse-lookup dance.

## Problem Statement

**Structural problem (feedback).** The PRD → sources edge in the kiln data model is stored only in a rendered markdown table. Every consumer that needs to traverse that edge must either reparse markdown or do an O(N) scan over the backlog to invert the reverse edge. This is architectural drift: we have a perfectly good frontmatter schema on both endpoints (PRDs and backlog items) and we are using it only for one direction of a bidirectional relationship. The drift stays invisible until a consumer adds nontrivial logic to do the reverse lookup — and then the logic becomes a bug magnet because it has to handle path-normalization edge cases the source-of-truth edge doesn't.

**Tactical evidence.** PR #146 (pipeline-input-completeness) shipped a fix for exactly one manifestation of this drift: `/kiln:kiln-build-prd` Step 4b was scanning only `.kiln/issues/` and not `.kiln/feedback/`, and its path-matching logic couldn't handle trailing slashes or absolute-vs-relative paths. The root-cause retro note for that fix observed that the "string-match pattern keeps costing us real bugs" — PR #146 was the second time this year. The R-1 "bless strict behavioral superset" precedent (hygiene rubric's two applications) is another signal that the string-match implementation is fragile under small-scale refactors. `/kiln:kiln-hygiene`'s `merged-prd-not-archived` rule contains its own copy of the same inversion logic and is therefore exposed to the same fragility on any future refactor.

This PRD is not about fixing a bug — PR #146 already fixed the immediate bug. It is about removing the structural precondition that makes the same class of bug cheap to write.

## Goals

- **Make the PRD → sources edge machine-readable.** Every PRD generated by `/kiln:kiln-distill` carries a `derived_from:` list in YAML frontmatter listing every source item (relative paths, feedback first per FR-012), plus `distilled_date:` and `theme:` for traceability.
- **Switch consumers to read-PRD-once.** `/kiln:kiln-build-prd` Step 4b and `/kiln:kiln-hygiene`'s `merged-prd-not-archived` rule both switch from scan-and-match to "read the PRD, iterate `derived_from`." The 6-field Step 4b diagnostic from PR #146 stays in place (for visibility and backwards-compat on PRDs missing `derived_from:`), but on a PRD with frontmatter present the matched-count invariant should always equal the list length.
- **Establish the body table as a render of frontmatter.** The `### Source Issues` markdown table becomes a render of `derived_from:` — frontmatter is the single source of truth; the table is for human readability. The two must not be allowed to drift.
- **Migrate existing PRDs.** Ship a one-shot migration that backfills `derived_from:` on PRDs under `docs/features/` by parsing their Source Issues table. Follows the propose-don't-apply pattern established by `/kiln:kiln-claude-audit` and `/kiln:kiln-hygiene` — writes a diff preview, doesn't apply.

## Non-Goals

- **No change to the `.kiln/feedback/` or `.kiln/issues/` frontmatter schema.** The `prd:` field on source items is already correct and stays as-is.
- **No removal of the 6-field Step 4b diagnostic from PR #146.** That diagnostic remains as a backwards-compat guard rail for any PRD that lacks `derived_from:` (pre-migration) or for any future distill path that forgets to write it. The goal is to make the frontmatter path the primary code path, not to delete the fallback.
- **No new schema for `distilled_date:` / `theme:` beyond what goes in the PRD.** These are purely metadata fields — no consumer is required to read them in this PRD's scope.
- **No propagation of `derived_from:` into `docs/PRD.md` or `products/<slug>/PRD.md`.** Those are product-level PRDs not produced by distill; they have no source items.
- **No policy engine or "all frontmatter fields must be used" audit.** Just this one field added for this one concrete payoff.

## Requirements

### Functional Requirements

**Distill-side (writer):**

- **FR-001 (from `.kiln/feedback/2026-04-24-prds-generated-by-kiln-kiln-distill-should-carry.md`)**: `/kiln:kiln-distill` Step 4, when generating a feature PRD, MUST write YAML frontmatter at the top of the PRD file containing the keys `derived_from:` (list of relative paths from repo root, one entry per source item bundled into the PRD), `distilled_date:` (ISO 8601 date when distill ran), and `theme:` (the theme slug used for the PRD directory). Frontmatter keys are written in this exact order. Feedback-source paths appear BEFORE issue-source paths within `derived_from:` (matches FR-012 ordering).
- **FR-002 (from same)**: The `### Source Issues` markdown table in the PRD body MUST be rendered from the `derived_from:` frontmatter list — same order, same entries. `/kiln:kiln-distill` writes both from a single in-memory list so they cannot drift at write time.
- **FR-003 (from same)**: The `derived_from:` paths MUST be repo-relative and use forward slashes regardless of host OS. Paths are the on-disk path of the source file at distill time (not the Obsidian path, not the GitHub URL).

**Build-prd consumer (Step 4b):**

- **FR-004 (from same)**: `/kiln:kiln-build-prd` Step 4b MUST, as its primary path, read the current PRD's `derived_from:` frontmatter and iterate that list. For each entry, it flips `status: prd-created` → `status: completed`, adds `completed_date:` and `pr:`, and moves the file into the `completed/` subdirectory of `.kiln/issues/` or `.kiln/feedback/` as appropriate.
- **FR-005 (from same)**: If the PRD being built has no `derived_from:` frontmatter (pre-migration PRD), Step 4b falls back to the existing PR-#146 scan-and-match behavior with its 6-field diagnostic. The diagnostic line's `prd_path` field is unchanged; an additional field `derived_from_source:` reports either `frontmatter` or `scan-fallback`.
- **FR-006 (from same)**: When reading from `derived_from:`, if any entry in the list does NOT exist on disk at the stated path, Step 4b records the missing entry in the diagnostic line (new field `missing_entries:`), continues processing the remaining entries, and does NOT fail the pipeline. The matched-count invariant (matched == len(derived_from)) fires ONLY when `missing_entries` is 0.

**Hygiene consumer (`merged-prd-not-archived`):**

- **FR-007 (from same)**: `plugin-kiln/rubrics/structural-hygiene.md`'s `merged-prd-not-archived` rule MUST, as its primary path, read each PRD's `derived_from:` frontmatter and emit one `archive-candidate` / `needs-review` / `inconclusive` signal per entry in the list (based on the entry's current `status:` on disk). The existing bulk `gh pr list` call still populates the merged-PR map, but the reverse-lookup step changes from "walk backlog, string-match `prd:` against merged PRDs" to "walk PRDs, read `derived_from:`, look up PR merge state."
- **FR-008 (from same)**: PRDs without `derived_from:` (pre-migration) still get processed by the existing walk-backlog path. The skill continues to scan `.kiln/issues/*.md` and `.kiln/feedback/*.md` for any item whose `prd:` frontmatter points at a PRD lacking `derived_from:`. This preserves today's behavior for unmigrated PRDs.

**Migration:**

- **FR-009 (from same)**: A one-shot migration skill or script MUST be shipped that, for every PRD under `docs/features/<date>-<slug>/PRD.md` and `products/<slug>/features/<date>-<slug>/PRD.md`, parses the body's `### Source Issues` table (if present), composes a `derived_from:` frontmatter block, and writes a review preview diff under `.kiln/logs/prd-derived-from-backfill-<timestamp>.md`. The migration follows the propose-don't-apply discipline of `/kiln:kiln-claude-audit` and `/kiln:kiln-hygiene` — it writes a diff preview, it does NOT call Edit/Write against any PRD file.
- **FR-010 (from same)**: The migration MUST be idempotent: a PRD that already has `derived_from:` in its frontmatter is skipped (no diff emitted for it).
- **FR-011 (from same)**: For PRDs under `products/` (clay multi-product repo shape) the migration follows the same shape and writes the same diff preview. It does not special-case the product-directory path layout.

### Non-Functional Requirements

- **NFR-001**: Backwards compatibility. Existing PRDs (pre-migration) continue to work end-to-end — Step 4b and hygiene both fall back to scan-and-match when `derived_from:` is absent. Pipelines mid-flight when the update lands do not break.
- **NFR-002**: Plugin portability. All three consumer changes live under `plugin-kiln/`. No repo-relative scripts are introduced in command-step workflows (per CLAUDE.md plugin workflow portability rule). The migration tool, if implemented as a script, must be resolvable via `${WORKFLOW_PLUGIN_DIR}/scripts/...` if invoked from a wheel workflow, or live as an inline `SKILL.md` body if invoked as a `/skill` command.
- **NFR-003**: Idempotence and determinism. All three writer/reader paths must be idempotent under repeated invocation on unchanged input — a second run of distill on the same inputs writes byte-identical `derived_from:` (sorted within each source-type group by filename ASC); a second Step 4b run after successful archival is a no-op; a second migration run emits zero diffs.
- **NFR-004**: No new MCP calls. The migration and all three consumer changes are pure local-file operations. No Obsidian writes, no GitHub API calls beyond what's already in the hygiene rule.
- **NFR-005**: Diagnostic continuity. The Step 4b 6-field diagnostic from PR #146 MUST continue to be emitted on every run. The two new fields (`derived_from_source:`, `missing_entries:`) are additive — existing grep-anchored test patterns (if any) MUST continue to match.

## User Stories

**Story 1 — Pipeline author runs build-prd and items flip cleanly.**
As a maintainer who just ran `/kiln:kiln-build-prd <slug>`, I want every source item in the PRD's `derived_from:` list to be flipped to `completed` and archived — without the tool having to guess which items are in scope. Acceptance: a pipeline on a PRD with `derived_from:` present writes a diagnostic line with `derived_from_source: frontmatter`, `missing_entries: 0`, and `matched == len(derived_from)`. All listed items end the pipeline at `status: completed` in their respective `completed/` directories.

**Story 2 — Hygiene audit reads PRDs directly.**
As a maintainer who just ran `/kiln:kiln-hygiene`, I want the `merged-prd-not-archived` rule to read each PRD's `derived_from:` once instead of walking every backlog file. Acceptance: the hygiene audit's preview shows one archive-candidate signal per entry in each migrated PRD's `derived_from:`. For pre-migration PRDs, the walk-backlog fallback still fires and produces identical output to today.

**Story 3 — Existing PRDs get backfilled in one pass.**
As a maintainer with 20+ existing PRDs under `docs/features/`, I want a single invocation to propose a `derived_from:` backfill for all of them so I can review and apply the diff manually. Acceptance: `/kiln:kiln-prd-backfill` (or equivalent) writes `.kiln/logs/prd-derived-from-backfill-<timestamp>.md` containing one unified-diff hunk per PRD missing `derived_from:`. A second invocation on the same state emits zero hunks.

## Success Criteria

- **SC-001**: `/kiln:kiln-distill` produces a PRD whose YAML frontmatter contains `derived_from:`, `distilled_date:`, and `theme:` keys in that order. Verified by running `/kiln:kiln-distill` on a test-fixture backlog and inspecting `docs/features/<date>-<slug>/PRD.md` frontmatter.
- **SC-002**: `/kiln:kiln-build-prd` Step 4b, running on a PRD with `derived_from:` present, flips every listed item to `completed` and archives it. Verified by running a full `/kiln:kiln-build-prd` pipeline end-to-end and inspecting the diagnostic line plus `.kiln/issues/completed/` and `.kiln/feedback/completed/` after the pipeline completes.
- **SC-003**: `/kiln:kiln-build-prd` Step 4b, running on a PRD WITHOUT `derived_from:` (legacy PRD), continues to work via the PR-#146 scan-and-match fallback. Verified by pointing a pipeline at a pre-migration PRD and confirming the diagnostic line reports `derived_from_source: scan-fallback` and completes archival successfully.
- **SC-004**: `/kiln:kiln-hygiene`'s `merged-prd-not-archived` rule emits correct signals for both migrated and unmigrated PRDs in a single run. Verified by running the hygiene skill on a mixed-state fixture and diffing against expected output.
- **SC-005**: The backfill migration, run twice on the same repo, emits zero diffs on the second run. Verified by running twice and inspecting both log files.
- **SC-006**: The `### Source Issues` table in the PRD body matches the frontmatter `derived_from:` list byte-for-byte in path order. Verified by a diff check inside `/kiln:kiln-distill`'s smoke fixture.
- **SC-007**: Backwards compat — running `/kiln:kiln-build-prd` on `docs/features/2026-04-23-pipeline-input-completeness/PRD.md` (just-merged PR #146, no `derived_from:`) still completes successfully. Verified by running the pipeline in a verification branch (or via the backfill migration applied first, then re-run).
- **SC-008**: Documentary smoke fixture under `specs/prd-derived-from-frontmatter/SMOKE.md` includes: (a) a before/after `derived_from:` frontmatter sample, (b) a Step 4b diagnostic line with both new fields present, (c) a migration diff hunk sample.

## Risks & Open Questions

- **Risk 1 — Migration false positives.** The backfill parses the `### Source Issues` table as-is. If any existing PRD has a malformed or hand-edited table, the migration could backfill the wrong paths. Mitigation: migration writes a diff preview, doesn't apply; human review is the gate. Additional guard: migration should validate that each candidate `derived_from:` entry exists on disk at the parsed path, and flag mismatches in the preview.
- **Risk 2 — Drift between frontmatter and body table post-distill.** If a human hand-edits the `### Source Issues` body table but doesn't update `derived_from:`, consumers will read the frontmatter and ignore the body edit. Mitigation: documented in CLAUDE.md that frontmatter is authoritative; distill's smoke fixture asserts SC-006 on every distill run; `/kiln:kiln-hygiene` could optionally add a rule checking the two are in sync (deferred to a follow-on PRD, not in scope here).
- **Risk 3 — Path format drift.** Repos mixing absolute paths, relative paths, and case variations (Windows-like) could produce inconsistent `derived_from:` entries. Mitigation: FR-003 fixes format at write time (repo-relative, forward-slash, canonical case). Consumers should not need to normalize when reading from `derived_from:` — any normalization is a bug in distill.
- **Open question 1**: Should the migration be a standalone `/kiln:kiln-prd-backfill` skill, or a subcommand of `/kiln:kiln-hygiene`? The latter consolidates propose-don't-apply audits under one entry point; the former is a one-shot that makes its lifecycle cleaner. Plan phase decision.
- **Open question 2**: Should `distilled_date:` be UTC or local? Recommend UTC for determinism, matching `/kiln:kiln-feedback`'s `date:` convention.
- **Open question 3**: When a PRD is hand-authored (not via `/kiln:kiln-distill`), should it carry `derived_from:` at all? For `docs/PRD.md` and `products/<slug>/PRD.md` — no (Non-Goals). For feature PRDs hand-written by a maintainer without going through distill — recommend adding the block manually with `derived_from: []` or omitting it entirely; Step 4b's diagnostic handles both cases. Plan phase decision.
