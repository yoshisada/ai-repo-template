# Speckit Reset — Clear Feature Artifacts

## User Input

```text
$ARGUMENTS
```

You **MUST** consider the user input before proceeding (if not empty).

## Overview

This skill removes all artifacts generated by a speckit pipeline run for a feature, so you can re-run the pipeline from scratch. It supports three reset levels depending on how much you want to clear.

## Step 1: Detect the Feature

1. Run `.specify/scripts/bash/check-prerequisites.sh --paths-only --json` to get `FEATURE_DIR` and `BRANCH`.
   - If this fails (e.g., not on a feature branch), check if the user provided a feature name in `$ARGUMENTS` and look for `specs/<feature-name>/`.
   - If no feature can be detected, ERROR: "No feature detected. Run from a feature branch or pass the feature name as an argument."

2. Read `specs/<feature>/plan.md` if it exists — extract the list of source directories that were created (look for file paths in the plan's project structure section). These are the implementation directories to remove.

3. Display what will be removed and ask for confirmation:

```
## Speckit Reset: {feature-name}

### Reset Levels

**Level 1 — Specs only** (re-run from /speckit.specify):
- Remove: specs/{feature}/ (spec.md, plan.md, tasks.md, contracts/, research.md, data-model.md, quickstart.md, blockers.md, checklists/)

**Level 2 — Specs + implementation** (re-run full pipeline):
- Remove: specs/{feature}/
- Remove: {detected source dirs from plan.md, e.g., cli/, templates/, modules/}
- Remove: vendor/ (if researcher created it)

**Level 3 — Full branch reset** (start completely fresh):
- Git reset to the commit before any speckit work on this branch
- Equivalent to: git log to find the first speckit commit, then reset to its parent

Which level? (1/2/3)
```

## Step 2: Execute Reset

### Level 1 — Specs Only

```bash
rm -rf specs/{feature}/
```

Preserves all implementation code. Use this when specs need rework but code is fine, or when you want to re-specify without losing implementation progress.

### Level 2 — Specs + Implementation

1. Remove spec artifacts:
   ```bash
   rm -rf specs/{feature}/
   ```

2. Detect and remove implementation directories. Use plan.md (if it was readable in Step 1) to identify source dirs. Common patterns:
   - `cli/` or `src/` — source code
   - `templates/` — template files
   - `modules/` — module definitions
   - `vendor/` — researcher-cloned dependencies
   - `tests/` — test files (only if created by this feature, not pre-existing)

   **Safety check**: Before removing each directory, verify it was created by this pipeline run:
   - Check `git log --diff-filter=A --name-only` to see if the directory was added on this branch
   - Do NOT remove directories that existed before this feature branch was created
   - If uncertain, list the directory and ask the user to confirm removal

3. Remove generated lockfiles and build artifacts:
   ```bash
   rm -rf node_modules/ bun.lock coverage/ .turbo/
   ```
   Only remove these if they are inside directories being cleaned (not root-level unless the whole project is generated).

### Level 3 — Full Branch Reset

1. Find the first speckit commit on this branch:
   ```bash
   # Find where this branch diverged from main
   MERGE_BASE=$(git merge-base main HEAD)
   ```

2. Show the user what will be reset:
   ```bash
   git log --oneline $MERGE_BASE..HEAD
   ```

3. After user confirms, reset:
   ```bash
   git reset --hard $MERGE_BASE
   ```

4. The branch is now clean — ready for a fresh `/speckit.run`.

## Step 3: Verify and Report

After reset, verify the state:

```
## Reset Complete: {feature-name}

**Level**: {1/2/3}
**Removed**:
- specs/{feature}/ — {file count} files
- {other dirs if level 2+}

**Preserved**:
- docs/PRD.md — your requirements document
- .specify/ — speckit configuration and templates
- .claude/ — skills and hooks
- CLAUDE.md — project instructions

**Next steps**:
- Level 1: Run `/speckit.specify` to re-create specs
- Level 2: Run `/speckit.run` to re-run the full pipeline
- Level 3: Run `/speckit.run` to start fresh
```

## Safety Rules

- **NEVER remove** `docs/PRD.md`, `.specify/`, `.claude/`, `CLAUDE.md`, or `.git/`
- **NEVER remove** files that existed before the feature branch was created
- **ALWAYS ask for confirmation** before removing anything — show the file list first
- **Level 3 is destructive** — warn the user that uncommitted changes will be lost
- If `$ARGUMENTS` contains `--dry-run`, show what would be removed without removing anything
- If `$ARGUMENTS` contains `--level 1|2|3`, skip the prompt and use the specified level
- If `$ARGUMENTS` contains `--yes` or `-y`, skip confirmation (for scripted usage)
