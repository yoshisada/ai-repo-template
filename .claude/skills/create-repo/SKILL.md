# Create Repo — Scaffold a New Project from Template

Create a new GitHub repository with the full ai-repo-template infrastructure (speckit workflow, hooks, skills, agents, constitution, templates) and optionally seed it with PRD artifacts.

## User Input

```text
$ARGUMENTS
```

You **MUST** consider the user input before proceeding (if not empty).

## Step 1: Gather Information

If not provided in user input, ask these questions in **one message**:

1. **Repo name**: What should the repository be called? (e.g., `my-saas-app`, `billing-service`)
2. **GitHub org or user**: Where should it live? (default: current authenticated user via `gh api user --jq .login`)
3. **Visibility**: Public or private? (default: private)
4. **Description**: One-line repo description
5. **PRD seeding**: Should we copy PRD artifacts into the new repo?
   - If `docs/PRD.md` has real content (not the placeholder template): offer to copy it
   - If `docs/features/*/PRD.md` exists: offer to copy a specific feature PRD
   - If no PRDs exist: skip this — user can run `/create-prd` in the new repo later
6. **Local path**: Where to clone locally? (default: sibling directory `../<repo-name>`)

## Step 2: Create the GitHub Repository

```bash
gh repo create <owner>/<repo-name> --<visibility> --description "<description>" --clone
```

If the repo already exists, stop and ask the user how to proceed. Do NOT overwrite.

## Step 3: Scaffold the Template

Copy the template infrastructure from this repo (`yoshisada/ai-repo-template`) into the new repo. The source is the current working directory.

### Files to copy (preserving directory structure)

**Core workflow:**
- `CLAUDE.md`
- `.gitignore`
- `scripts/setup.sh`

**Claude Code configuration:**
- `.claude/settings.json`
- `.claude/hooks/require-spec.sh`
- `.claude/hooks/block-env-commit.sh`
- `.claude/agents/` — all agent files
- `.claude/commands/` — all command files
- `.claude/skills/` — all skill directories (including this one)

**Speckit infrastructure:**
- `.specify/init-options.json`
- `.specify/memory/constitution.md`
- `.specify/templates/` — all template files
- `.specify/scripts/` — all script files

**Documentation structure:**
- `docs/session-prompt.md`
- `docs/PRD.md` — copy the **placeholder template** (not the filled-in PRD from this repo, unless seeding)
- `specs/README.md`

**Empty directories (create with .gitkeep):**
- `src/.gitkeep`
- `tests/.gitkeep`

### Copy method

```bash
# From the new repo directory
rsync -a --exclude='.git' --exclude='node_modules' --exclude='dist' \
  --exclude='docs/features' \
  --include='.claude/***' \
  --include='.specify/***' \
  --include='.gitignore' \
  --include='CLAUDE.md' \
  --include='scripts/***' \
  --include='docs/session-prompt.md' \
  --include='docs/PRD.md' \
  --include='specs/README.md' \
  <template-dir>/ ./
```

Alternatively, copy files individually if rsync filtering is complex. The goal is to get all infrastructure files without any project-specific content from the template repo.

### Files to NOT copy
- `docs/features/` — feature PRDs are project-specific (unless seeding)
- `specs/*/` — existing feature specs are project-specific
- Any `node_modules/`, `dist/`, `build/`, `.next/`, `.expo/`
- `.env` files
- `.git/` directory
- `.mcp.json` — will be regenerated by setup.sh

## Step 4: Seed PRD Artifacts (if requested)

If the user chose to seed PRD artifacts in Step 1:

### Seeding a product PRD
Copy from this repo to the new repo:
- `docs/PRD.md` (the real one, overwriting the placeholder)
- `docs/PRD-MVP.md` (if exists)
- `docs/PRD-Phases.md` (if exists)

### Seeding a feature PRD
Copy from this repo to the new repo:
- `docs/features/<selected-feature>/PRD.md` → `docs/PRD.md` in the new repo (promote feature PRD to product PRD)

OR if the user wants it as a feature in the new repo too:
- `docs/features/<selected-feature>/` → `docs/features/<selected-feature>/` in the new repo

Ask the user which approach they prefer.

## Step 5: Run Setup

```bash
cd <new-repo-path>
chmod +x scripts/setup.sh
bash scripts/setup.sh
```

This installs speckit, configures MCP, makes hooks executable, and verifies the scaffold.

If setup.sh fails on any step, report the error and suggest manual fixes. Do NOT silently continue.

## Step 6: Initial Commit and Push

```bash
cd <new-repo-path>
git add -A
git commit -m "Initial scaffold from ai-repo-template

Includes: speckit workflow, 4-gate hooks, skills, agents, constitution,
templates, and setup infrastructure.

Source: yoshisada/ai-repo-template"
git push -u origin main
```

## Submodule / Monorepo Mode

If the user says "create a submodule", "add a module", or the target path is inside an existing repo:

1. Do NOT create a new GitHub repo
2. Instead, scaffold the template into a subdirectory of the current repo
3. Skip `.git` creation — use the parent repo's git
4. Adjust paths in `CLAUDE.md` and hook scripts to be relative to the subdirectory
5. Add the subdirectory to the parent repo's `.gitignore` if it should be independent, or commit it if it's part of the monorepo
6. Skip `gh repo create` and `git push` — the parent repo manages that

Ask the user: "Should this module be a git submodule (independent repo linked here) or a subdirectory (part of this repo)?"

### Git submodule path
```bash
# Create the repo first (Steps 2-6 above), then:
cd <parent-repo>
git submodule add https://github.com/<owner>/<repo-name> <path>
git commit -m "Add <repo-name> as submodule at <path>"
```

### Subdirectory path
```bash
# Scaffold directly into the subdirectory, skip repo creation
mkdir -p <path>
# Copy template files into <path>/
# Commit as part of parent repo
```
