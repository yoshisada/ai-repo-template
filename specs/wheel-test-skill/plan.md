# Implementation Plan: Wheel Test Skill

**Branch**: `build/wheel-test-skill-20260410` | **Date**: 2026-04-10 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/wheel-test-skill/spec.md`

## Summary

Ship a new user-invocable Claude Code skill, `/wheel-test`, that executes every JSON workflow under `workflows/tests/` end-to-end and produces a markdown pass/fail report at `.wheel/logs/test-run-<timestamp>.md`. The skill is a SKILL.md file (Markdown with inline Bash) living at `plugin-wheel/skills/wheel-test/SKILL.md`. It classifies workflows into 4 phases based on JSON step types, activates Phase 1 back-to-back in quick succession for safe parallelism, and runs Phases 2–4 serially. All workflow progression goes through the existing `plugin-wheel/bin/activate.sh` + hook contract — the skill never writes state files or advances cursors directly.

## Technical Context

**Language/Version**: Markdown (skill definition), Bash 5.x (inline shell commands invoked by the skill), `jq` (JSON inspection)
**Primary Dependencies**: `plugin-wheel/bin/activate.sh`, `plugin-wheel/bin/validate-workflow.sh`, existing wheel hook contracts (`plugin-wheel/hooks/*`), `jq`
**Storage**: Filesystem only — reads `workflows/tests/*.json`, inspects `.wheel/state_*.json`, watches `.wheel/history/{success,failure,stopped}/`, writes `.wheel/logs/test-run-<UTC-timestamp>.md`
**Testing**: Manual smoke-test via running the skill against the real test suite in-repo (shell-script unit tests under `tests/*.sh` are out of scope per PRD non-goals). Primary validation is that the skill itself passes against the current 12-workflow suite.
**Target Platform**: Developer machines running Claude Code with the kiln+wheel plugins installed (macOS / Linux, Bash 5.x)
**Project Type**: Claude Code plugin skill (Markdown + inline Bash). No compiled artifact, no dependency install step.
**Performance Goals**: Full 12-workflow run under 5 minutes wall clock (SC-001). Phase 1 activations back-to-back (no gating between activations).
**Constraints**: No new language runtime. No Node.js. No mocks. No bypassing hooks. Must leave `.wheel/state_*.json` empty after a passing run.
**Scale/Scope**: Current suite is 12 workflows; design must tolerate growth without code changes (FR-001 auto-discovery).

## Phase Execution Model

The skill orchestrates the test run in 5 sequential passes:

1. **Pre-flight** (FR-007, FR-014): Verify `workflows/tests/` is non-empty, verify `.wheel/state_*.json` is empty, record baseline line count of `.wheel/logs/wheel.log` (FR-009), record start timestamp.
2. **Classify** (FR-001, FR-002): For each `workflows/tests/*.json`, use `jq` to extract the set of step types and assign a phase (1/2/3/4). Classification is JSON-driven, not filename-driven.
3. **Run phases in order** (FR-003, FR-004, FR-005, FR-006, FR-015):
   - **Phase 1**: Call `activate.sh` for every Phase 1 workflow back-to-back (each activation a separate Bash tool call). Wait up to 60 seconds per workflow for its archive to appear. After all phase 1 workflows archive, check for orphans (FR-008).
   - **Phase 2**: One workflow at a time. Activate, wait ≤60s for archive, orphan check, move to next.
   - **Phase 3**: Same as Phase 2.
   - **Phase 4**: One workflow at a time. Follow the stop-hook flow exactly (wait for TeamCreate instruction, TeamCreate, spawn teammates with `run_in_background: true` via Agent tool, wait for teammate results, send shutdown_requests, wait for teammate_terminated, TeamDelete). Timeout ≤120s per workflow.
4. **Verify archives + collect hook errors** (FR-010, FR-009, FR-018): For each workflow, confirm the archive file exists with the hybrid `{workflow}-{ts}-{stateid}.json` format under `history/success/` or `history/failure/` (or `history/stopped/` → treated as failure). Scan `.wheel/logs/wheel.log` from the recorded baseline line count onward for `ERROR|FAIL|stalled`.
5. **Report** (FR-011, FR-012, FR-013): Build the markdown report with header/summary/per-workflow table/orphans/hook-errors/repro-commands. Write to `.wheel/logs/test-run-<UTC-timestamp>.md` and echo to stdout. Print the report path. Print the final verdict line (`PASS` or `FAIL (M failed, K orphaned, L hook errors)`).

## Open Decisions (Resolved)

Taken directly from the PRD's Open Questions and resolved during planning:

- **Q1 (verbose/quiet mode)** → No. The report has all the detail; add later if needed.
- **Q2 (phase 1 early-finishers)** → All Phase 1 workflows must archive before Phase 2 starts. Simpler reasoning, negligible cost.
- **Q3 (stopped-archive outcome)** → Failure with "stopped unexpectedly" note. Encoded as FR-018.
- **Q4 (auto-archive orphans at pre-flight)** → Refuse to proceed. Auto-archiving could mask a real problem.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Phase 1 race on same-second activation | Rely on the hybrid archive filename (commit `69d2dff`) which includes state id suffix. |
| Team workflow shutdown latency | 120s Phase 4 timeout accommodates current worst case (~30s from issue #007). |
| Hook assumes one active workflow at a time | Phase 1 intentionally exercises the path; failures are a feature — they catch the assumption. |
| `.wheel/logs/` not gitignored | Task includes verifying `.gitignore` and adding the entry if missing. |
| Invoker skips stop-hook ceremony for Phase 4 | SKILL.md documents each step verbatim and instructs the invoker to wait for each notification before acting. |
| Inline SKILL.md exceeds 500 lines | Extract helpers into `plugin-wheel/skills/wheel-test/lib/*.sh` during implementation if Principle VI is breached. |
