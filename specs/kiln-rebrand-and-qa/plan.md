# Implementation Plan: Kiln Rebrand, Infrastructure & QA Reliability

**Branch**: `build/kiln-rebrand-and-qa-20260331` | **Date**: 2026-03-31 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/kiln-rebrand-and-qa/spec.md`

## Summary

Rename the plugin from "speckit-harness" to "kiln" across all user-facing surfaces (npm package, plugin manifest, skill names, documentation, scaffold templates). Establish `.kiln/` as the standard directory for automation artifacts in consumer projects. Add a `/kiln-doctor` skill for state validation and legacy path migration. Fix QA reliability by adding build version verification pre-flight checks.

## Technical Context

**Language/Version**: JavaScript (Node.js ESM) + Bash shell scripts
**Primary Dependencies**: Claude Code plugin system (skills, agents, hooks auto-discovery)
**Storage**: File system (markdown, JSON, shell scripts) — no database
**Testing**: Pipeline validation via `/build-prd` on consumer projects (no unit test framework for the plugin itself)
**Target Platform**: macOS, Linux (Claude Code CLI environments)
**Project Type**: Claude Code plugin (npm package)
**Performance Goals**: Doctor scan < 10 seconds, QA version check < 30 seconds additional pre-flight time
**Constraints**: Backwards compatible with existing consumer projects; idempotent operations; no partial rename states
**Scale/Scope**: ~30 skill directories, 8 agent files, 4 hooks, 8 templates, 1 scaffold set, 1 init script

**Doctor modes**:
- Diagnose: read manifest, compare against project state, report differences
- Fix: for each issue, present fix and apply on confirmation; idempotent

### Phase 4: QA Build Verification (FR-021 through FR-024)

**Files to modify**:
- `plugin/agents/qa-engineer.md` — add version verification pre-flight section
- `plugin/skills/qa-pass/SKILL.md` — add version check pre-flight step
- `plugin/skills/ux-evaluate/SKILL.md` — add version check pre-flight step

**Version check logic** (to be documented in agent/skill instructions):
1. Read VERSION file from project root
2. Check app for version display (UI, CLI output, build manifest)
3. Compare versions
4. On mismatch: trigger build command, re-check
5. On persistent mismatch: warn team lead, add disclaimer to report
6. Fallback: if no version in UI, check git SHA or build timestamp
